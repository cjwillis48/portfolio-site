# Code Review — Charlie Willis Portfolio

**Reviewer:** Claude (Opus 4.6)
**Date:** 2026-03-03
**Stack:** SvelteKit 2 + Svelte 5, Tailwind CSS 3, Cloudflare Pages, TypeScript

---

## Overall Impression

This is a clean, well-structured portfolio. The code reads like it was written by someone who actually
ships production software — not the typical framework-tutorial-pasted-together portfolio you usually see.
The Chat component alone is more sophisticated than most people's entire sites. Good separation of
concerns, proper error handling, solid security posture on the contact form.

**Grade: B+** — Strong fundamentals, a few things worth tightening up.

---

## What's Done Well

### Architecture
- **Lazy-loaded Chat component** with triple-SSR guard (dynamic import + `onMount` + `browser` check) shows
  you learned the hard way and solved it properly. The git history confirms this.
- **Static data modules** instead of a CMS is the right call for a portfolio this size. Zero runtime data
  fetching for pages = fast.
- **RAGR proxy pattern** — not exposing the upstream service URL to the client is correct. The proxy
  validates, sanitizes, and maps errors before forwarding. This is how you do it.
- **Clean route structure** — file-based routing used as intended, no over-nesting.

### Security
- Contact form has **three layers of bot protection**: honeypot field, timing validation (3s minimum),
  and silent acceptance for honeypot triggers (so bots don't learn what tripped them). Smart.
- Chat markdown is rendered through `marked` then sanitized with `DOMPurify`. The `@html` is justified here.
- Input validation on both client and server for the chat proxy (length limits, history truncation, slug regex).
- Server errors return generic messages — no stack traces or internals leaked to the client.

### Code Quality
- Consistent use of Svelte 5 runes (`$state`, `$props`) — no mixing of old/new reactivity APIs.
- TypeScript is used throughout with proper interfaces, not `any` soup.
- The SSE streaming parser in Chat.svelte is well-structured with proper buffer management and
  the `\n\n` block splitting.

## Issues

### Medium Priority

#### 1. `createRequestId()` is duplicated three times
`src/hooks.server.ts:4`, `src/routes/contact/+server.ts:11`, `src/routes/api/ragr/+server.ts:26`

Same function, same fallback logic, copy-pasted. Extract to a shared util.

```ts
// src/lib/utils/request.ts
export function createRequestId(): string { ... }
```

#### 2. `emptyStateHint` reuses `placeholder` instead of a dedicated field
`src/lib/components/Chat.svelte:140`

```ts
emptyStateHint = model.placeholder?.trim() || 'Ask about experience, projects, or skills.';
```

This reads `model.placeholder` for *both* `inputPlaceholder` (line 139) and `emptyStateHint` (line 140).
These are semantically different — the input placeholder is a prompt for the textarea, while the empty
state hint is guidance shown in the message area. If the upstream model ever provides a long placeholder,
it'll look odd as a hint above the messages. Probably should be a separate field or at least intentionally
documented as a shared value.

#### 3. Dark mode is hardcoded with no toggle
`src/app.html:2` — `<html lang="en" class="dark">`

The entire Tailwind config is set up with `darkMode: 'class'` and every component has both light and dark
variant classes written out. That's a lot of dead CSS for a mode the user can never reach. Two options:
- Add a theme toggle (and persist preference in localStorage)
- Remove all the `dark:` prefixed classes and just style for dark directly — saves bundle size

The current state is the worst of both worlds: maintaining two themes but only showing one.

#### 4. Contact form `formStartedAt` is trivially spoofable
`src/routes/contact/+page.svelte:6` — `let formStartedAt = $state(Date.now())`

The timestamp is generated client-side and sent in the POST body. A bot can just send
`formStartedAt: Date.now() - 60000`. The timing check only stops the laziest bots. Not a major issue
since you have the honeypot too, but worth noting that this isn't real protection — it's a speed bump.
A server-side token (set via a cookie or hidden field populated by a server load function) would be
harder to fake.

#### 5. No `maxlength` on contact form inputs
`src/routes/contact/+page.svelte:70-101`

The server validates `name <= 120`, `email <= 254`, `message <= 5000`, but the client-side form has no
`maxlength` attributes on the name or email inputs (the message textarea also lacks one). Users can type
freely and only discover the limit after submit fails. Add `maxlength` attributes to match server limits.

#### 6. Chat resize handle uses `isResizing` as a plain variable, not `$state`
`src/lib/components/Chat.svelte:53`

```ts
let isResizing = false;
```

This is a plain `let` while the rest of the component uses `$state()`. It works because `isResizing` is
only read inside event handlers (not the template), so reactivity isn't needed. But it's inconsistent and
will confuse future-you. Either make it `$state` for consistency, or add a comment explaining why it's
intentionally non-reactive.

### Low Priority

#### 7. Nav active-link detection doesn't handle nested routes
`src/lib/components/Nav.svelte:37`

```svelte
{$page?.url?.pathname === href ? 'active styles' : 'default styles'}
```

Exact match means `/projects/foo` wouldn't highlight "Projects" in the nav. Not a problem today since
there are no nested routes, but if you ever add project detail pages, this will silently break. A
`pathname.startsWith(href)` check (with special-casing for `/`) would be more resilient.

#### 8. No `error.html` fallback page
If SvelteKit itself fails to render (not just a route error), there's no static fallback. Cloudflare
will show its default error page. A simple `src/error.html` with your branding would be a nice touch.

#### 9. Blog link could use `target="_blank"` feedback
`src/routes/+page.svelte:52` and `Nav.svelte:34`

The blog link opens in a new tab. On desktop this is fine, but on mobile the user might not realize
they've left the site. An external link icon (↗) next to "Blog" in the nav would be a small UX win.

#### 10. Chat panel has no keyboard shortcut to close
The chat opens with a click, closes with a "Close" button. No `Escape` key handler. For a resizable
floating panel, `Escape` to close is expected behavior.

#### 11. No `aria-expanded` on mobile menu button
`src/lib/components/Nav.svelte:48-50`

The hamburger button toggles the menu but doesn't communicate state to screen readers. Add
`aria-expanded={menuOpen}` to the button element.

#### 12. The `whatIDo` data module is unused
`src/lib/data/whatIDo.ts` exists but is never imported anywhere. Either use it or delete it — dead
modules are confusing for anyone reading the codebase.

## Nitpicks (Take or Leave)

- **`ProjectCard.svelte`** wraps each project in a card but the cards aren't links — there's nowhere to
  click through for more detail. For a portfolio, this is a missed opportunity. Even if you don't have
  detail pages, making the title a link to the relevant company/project would add interactivity.
- **Hero section** (`+page.svelte`) — the availability badge ("Interested in exploring new opportunities")
  is hardcoded. If you stop looking, you'll forget to update it. Consider making this data-driven.
- **`experience.ts:76`** — the fallback slug `'grandma'` in `readChatConfig` is... interesting.
  `chatbotConfig.modelSlug` is `'chatlie'`, so this fallback probably never fires, but `'grandma'` as a
  default model slug would raise questions during an incident.
- **No OG/social meta tags** — the pages have `<title>` and `<meta name="description">` but no
  Open Graph or Twitter Card tags. Links shared on LinkedIn/Twitter will look plain.
- **No `404.svelte` page** — SvelteKit will render a generic error. A custom 404 with navigation back
  to the home page would be a better experience.

## Performance Notes

- The site is already fast by design — static data, no client-side data fetching for pages, lazy-loaded
  chat component.
- Images use `loading="lazy"` and `decoding="async"` — good.
- The hero image (`hero.webp`) loads eagerly (no `loading="lazy"`) which is correct since it's above the fold.
- Consider adding `fetchpriority="high"` to the hero image for a small LCP improvement.
- `marked` and `isomorphic-dompurify` are only needed for the chat widget, and they're already
  lazy-loaded via the dynamic Chat import. Nice.

---

## Summary

Solid portfolio codebase. The most impactful improvements would be:

1. Extract the duplicated `createRequestId` utility
2. Decide on dark mode (toggle or drop the unused light theme classes)
3. Add `aria-expanded` to the mobile nav toggle
4. Add a custom 404 page
5. Add OG meta tags for social sharing

None of these are blockers — the site works well as-is. The chat integration is genuinely impressive
and shows a level of craft that most portfolio sites don't have.
